## 3. 框架契約（不變量——任何擴充不得破壞）

1. **編號鐵律**：場次編號＝日期順序（第 1 場永遠最早）；任何日期變動後必呼叫 `renumber()`。
2. **個資邊界**：講者身分證字號／戶籍／帳號**不入系統**（領據留白手寫）；聯絡資料只存**私有 Google Sheet（學校 Workspace＝既有信任邊界）與使用者瀏覽器**，不進 git／公開面（v2.1 修訂 2026-07-10：自「只存瀏覽器」放寬至私有 Sheet，與否決 Supabase 的信任邊界一致）。
3. **匯入必經 `normalize()`**：外來 JSON 一律消毒，槽位固定 12、id 重生、日期驗格式。
4. **輸出必經預檢**：Hub 匯出過 `hubWarnings()`（對 talks.schema.json）；領據只含鐘點費（交通費實報實銷）。
5. **渲染安全**：所有使用者資料進 HTML 前過 `esc()`；事件一律 data-act 委派，不用 inline handler。
6. **單檔自足**：不引外部資源（離線、file:// 直開可用）——這也是 profile 內嵌而非外部 JSON 的原因（file:// 下 fetch 會被擋）。**適用範圍（v2.1 界定 2026-07-10）：v1 本機版與 `join.html`；v2 管理台＝託管 Web App，不受此條約束。**

## 5. v2 路線（請隊員評——本次審查重點）

**後端三案**（多助理協作、跨機同步）：
- **A. Google Sheets ＋ Apps Script**：學校 Workspace 內既有工具，多人即時、授課教師多半已熟悉；要寫同步層、離線弱。
- **B. 維持 localStorage ＋ JSON 備份交換**：零建設、單人穩；協作靠人工合併（現況）。
- **C. Hub repo 當後端**（talks.json 為真值、管理台讀寫 git）：與前台天然一致；但把行政資料（核銷進度）放公開 repo 不妥，需拆公開/私有兩層。

**裁決（2026-07-10，經兩位獨立審查者複核後定案）**：
- **v2 ＝ A 案＋「Sheet 真值＋Local-First 佇列」**（v2.1 修訂 2026-07-10，多人協作定案）：正式真值＝私有 Google Sheet；前端本地樂觀更新＋背景佇列 debounce 批次同步，**嚴禁同步阻塞 UI**——查證 GAS 併發上限 30、單請求延遲 1.5–3 秒；伺服器端以 per-record `updatedAt` 做衝突偵測，不得後寫覆蓋。
- **D 案（Supabase/Firebase BaaS）不採**：審查中被評為技術最優解（即時同步、離線佇列、權限），但被兩條契約否決——講者聯絡個資不外放第三方雲（學校 Google Workspace 是既有信任邊界）、CDN SDK 破壞單檔自足/離線契約。若未來需要真即時協作再重議。
- **C 案維持現狀**：只做單向匯出（隱私＋延遲不適合當行政後端）。

其餘 v2 項：gws-mail 一鍵建草稿（邀請/行前/感謝信）、期末成果報告一鍵彙整（吃 12 場的照片/簽到/學習單清單）、講者跨學期歷史。

## 5.1 框架審查後補強（2026-07-10 已落地）

- **LS_KEY 按課程隔離**：`talkmgr.v1.<courseDir>`，同瀏覽器可管多門課；舊單一鍵資料自動搬遷（審查修訂）。
- **normalize 全面 allowlist**：場次/講者/核銷/待辦逐欄位重建，外來備份的未知欄位（含誤存個資鍵）一律不落地；日期做格式＋真日期雙驗（審查修訂）。
- **契約自癒**：save() 前驗「編號＝日期順序」，被繞過自動 renumber＋console.warn（審查建議的輕量解，正常操作零誤觸發實測過）。
- **個資哨兵**：備份/複製前掃身分證字號 pattern，命中先攔問（審查建議的輕量版）。
- **defaultDB 只讀 profile**、Hub 提示文字吃 courseDir、週曆敘事字串進 profile（`weeksIntro`/`weeksAlert`）、`talkCount` 進 profile（審查修訂）。
- 領據補外籍未滿 183 天代扣 18% 警語；設定頁加「複製 Raw JSON」雲端手動備份路；download() 可帶 MIME（皆審查修訂）。

**backlog（先記不做）**：per-record `updatedAt`＋增量合併（v2 同步的前置）、待辦絕對日期覆寫、TALK_FIELDS 欄位宣告表、課程設定檔匯入精靈、共用 validateTalkCore()、Proxy 層級的日期監聽。

## 6. 已知限制與風險

- 單人單機（v1 設計如此）；快照防誤刪但防不了整個瀏覽器 profile 被清——備份 JSON 是最後防線。
- Hub schema 若改版，`hubWarnings` 預檢要跟著改（對接點只有 hubJson/hubWarnings 兩函式）。
- 主計室領據格式改版時，`buildReceiptBody` 要對新表。
- 檔案級複製的多課程策略：課多了會散——若超過 3 門課共用，再考慮 profile 陣列＋課程切換器。
- **領據產生器＝東華 adapter**：`buildReceiptBody` 忠實復刻東華主計室表格（中英標籤、欄位、對照表），跨校重用要整個換 adapter，這是刻意取捨、不是漏抽象（兩位審查者都點名，裁決：不為假想的跨校需求過度抽象——YAGNI）。
- 呈現層仍殘留少量課程敘事（指南表格文字等）——換課時讀一遍指南文案；完全 profile 化留給真的有第二門課那天。
- 備份 JSON 含講者聯絡資料（Email/電話）——檔案本身要當個資檔看待，別隨手傳（隱私由使用者端把關）。
